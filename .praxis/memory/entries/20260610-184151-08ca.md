title: Add systemic-complexity hotspot signal to /patterns instead of a new G-complexity gate
tags: architecture, systemic-complexity, patterns, refinement
---

**Context:** A review of praxis's doctrine for systemic-complexity coverage found
several real gaps (no continuous complexity/maintainability gate, no
architecture-fitness checks, no proactive detection of complexity accumulating
across many small changes, no Conway's-law/cross-repo complexity tracking, and
no self-audit of the harness's own process growth). praxis has run only one real
spec end-to-end so far.

**Decision:** Address only the proactive-detection gap, and only as a small
extension of existing tooling: `/patterns` (`tools/patterns.py`) now also reads
`implementation` ledger entries' "Files touched" lists and reports a file/module
recurring across 3+ entries as a "hotspot". The software-architect's
over-engineering practice (`references/practices.md` §10) gains a "Systemic
complexity" subsection explaining the boiling-frog problem and pointing hotspots
at `/refine`; the architecture review checklist gains item 8.6 to act on it when
re-reviewing a flagged area.

**Rationale:** The other gaps (a new mandatory `G-complexity` gate, automated
architecture fitness functions, periodic cumulative re-validation, Conway's-law
tracking, N-by-M cross-repo complexity management) would add new gates,
artifacts, or tooling without evidence that the problem has occurred in any
project run through praxis yet -- exactly the anticipation `promotion-policy.md`
and the operating model's anti-overbuild constraint warn against. The hotspot
signal, by contrast, reuses data the ledger already records (snapshot "Files
touched" lists), adds no new gate or artifact type, and routes to the existing
`/refine` lifecycle.

**Consequences:** Positive -- a systemic-complexity signal exists with zero new
process surface, and it is evidence-based by construction (nothing fires until
3+ real entries touch the same file). Negative -- it only catches complexity that
shows up as repeated edits to the same *file path*; cross-file/structural drift
(e.g. a dependency direction reversing across files) is not caught and remains a
deferred gap. Risks -- if recurrence is mostly doctrine/meta-churn (as seen on
first run, e.g. CHANGELOG.md, generated integration files) rather than product
code, the signal may need a path-based filter later; defer until evidence shows
it is noisy in a real product repo.
