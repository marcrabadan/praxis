## Context
User-supplied team conventions to bind into four role experts (developer, security-engineer, qa-engineer, software-architect). Raw form: (1) "no comments in code"; (2) "before complex logic, write unit tests / test-first". Existing coverage is generic best practice (developer practices.md: "comment the why, not the what"; descriptive TDD section) — the team wants stricter, binding rules. Org-specific because the team is choosing a stance the industry leaves open; capturable per skill-learner Rule 2 (user-supplied convention).

## Proposed rules (recommended formulation)

RULE 1 — Comment-free code (zero narrative comments).
Code must self-explain through names, small functions, and structure. Writing a comment to explain WHAT code does is a refactoring signal: rename/extract until the comment is unnecessary. The ONLY permitted comment states a constraint the code cannot express: a non-obvious invariant, an external/legal requirement, a deliberate trade-off, or a security warning — one line, stating WHY. Never: narrative comments, commented-out code, inline TODO/FIXME (track in tickets). Documented exceptions: docstrings on public/exported APIs are interface documentation, not narrative comments — they stay; design rationale lives in ADRs (`docs/decisions/`), never inline.

RULE 2 — Test-first for complex logic.
Before implementing complex logic, write the unit tests first (red → green → refactor). "Complex" is observable: branching/non-trivial conditionals, calculations (money, dates, rounding), parsing/serialization, state machines, concurrency, security-sensitive paths (authn/authz, validation, crypto), and any bug fix (regression test first — already doctrine in /fix-bug). Trivial glue/config/passthrough code may be test-with or test-after, but no complex logic merges without unit tests covering positive, negative, and boundary cases.

## Where each rule lands (smallest home; checklists are the enforcement hook — lifecycle subagents self-check against them)
- developer: tighten practices.md "Comments" + "Test-first/TDD" sections to the binding form; add checklist items (no narrative comments / no untested complex logic). Reconcile checklist line "Public APIs … have up-to-date docstrings or comments" with the docstring exception.
- qa-engineer: checklist item — verify complex logic carries unit tests written test-first (positive/negative/boundary), flag untested complexity as a release risk.
- security-engineer: checklist items — security-sensitive code is always "complex" (test-first with negative/abuse cases); no commented-out security controls; no secrets or workaround notes in comments.
- software-architect: practices/checklist — testability is a driving NFR (boundaries/DI so complex domain logic unit-tests in isolation); the WHY belongs in ADRs so code can stay comment-free.

## Consequences
Stricter than generic doctrine: comments become exceptional rather than discouraged; test-first becomes mandatory (not advisory) for complex logic. Enforced at review time via expert checklists and /review-changes. On accept, apply the edits to the four experts, run the skill validator on each, regenerate integrations, and commit.
