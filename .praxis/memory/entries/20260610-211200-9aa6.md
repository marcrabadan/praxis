## Context

User asked whether praxis's skills recognize the metrics SonarQube uses to
penalize code quality (Reliability/Security/Maintainability ratings,
Technical Debt Ratio, Coverage, Duplicated Lines, Cyclomatic/Cognitive
Complexity, Security Hotspots). A repo-wide search confirmed zero existing
references to SonarQube, quality gates, technical debt, or
cyclomatic/cognitive complexity anywhere in praxis doctrine. User asked to
add this and assign min/max allowed percentages.

## Decision

Add a new harness rule, `rules/code-quality-metrics.md`, giving a shared
vocabulary and SonarQube "Sonar way" default thresholds (Coverage on new code
>= 80%, Duplicated Lines on new code <= 3%, Maintainability/Reliability/
Security Rating = A on new code, Security Hotspots Reviewed = 100%,
Cyclomatic Complexity per function <= 10, Cognitive Complexity per function
<= 15, Technical Debt Ratio bands A<=5% .. E>50%). The rule states explicitly
that these are praxis defaults: a repo's own configured quality gate (Sonar
or equivalent) is binding when one exists, and these numbers/vocabulary apply
otherwise or as the shared language either way.

Wired as cross-references (plain repo-root paths, not relative markdown
links, since these files get bundled into integrations/codex|cursor|intellij
where relative paths would resolve incorrectly) from:
- developer/references/practices.md (DRY section) and checklist.md (items
  3.5 functions-do-one-thing / complexity, 3.6 duplication)
- qa-engineer/references/checklist.md (section B, new coverage-threshold item)
- software-architect/references/checklist.md (new item 8.7, alongside the
  8.6 hotspot item) and practices.md (new "Quantitative thresholds"
  subsection in §10)
- security-engineer/references/checklist.md (section 8 pipeline gates, new
  item ahead of G-security)

## Rationale

- Closes a real, confirmed gap (zero prior references) without inventing a
  new mandatory verify gate -- G-lint/G-tests/G-security already exist; this
  rule gives the skills preparing evidence for those gates concrete numbers
  and shared vocabulary.
- "Repo's configured gate wins" avoids the rule going stale or contradicting
  a team's actual customized Sonar profile.
- rules/ is the correct home per rules/source-of-truth.md ("rules/ -- ...
  and any future rule"); no registration elsewhere is required.

## Consequences

**Positive:** developer/qa-engineer/software-architect/security-engineer now
share one quantitative vocabulary for code-quality review; the new
maintainability-rating checklist item (8.7) gives the systemic-complexity
hotspot signal (8.6) a quantitative companion.

**Negative:** the default thresholds (80% coverage, 3% duplication, etc.) are
generic SonarQube defaults; if a project's real quality bar differs and has
no configured gate, these numbers could be cited as if they were binding when
they are only defaults.

**Risks:** none material -- read-only doctrine addition, no gate/workflow
changes, no new tooling.
