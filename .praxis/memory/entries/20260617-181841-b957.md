Add an optional, config-gated `deploy` phase to the feature-development lifecycle and the devops-engineer expert.

Decision (proposal, pending user accept):
- New optional terminal workflow step `deploy` after `release`, mirroring the optional `experience` step: it runs only when `.praxis/config.json` declares a `deploy` target, else it is skipped and recorded. Never assume a cloud target (never-assume rule applied to infra).
- Gate to enter: `release-approved` AND `deploy-target-configured`. Post-apply gate `deploy-healthy`; on failure routes back to `build` (forward fix). Artifact: reports/release/deploy-report.md.
- Targets: generic Kubernetes + AWS/EKS + GCP/GKE + Azure/AKS. Cloud-neutral workload manifests; only cluster/IAM/registry differ per provider.
- MCP as the execution backend: drive `terraform plan/apply` and cluster ops through a configured MCP server (recommended concrete servers documented: HashiCorp Terraform MCP, Kubernetes MCP, AWS/Azure/GCP MCPs); plan-only fallback when none is configured. praxis never hosts a server or holds credentials.
- Production apply always gates on a manual human promotion decision; rollback trigger defined before apply.
