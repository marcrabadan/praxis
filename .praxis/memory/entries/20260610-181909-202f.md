## Context
Refinement of accepted decision 20260610-181128-01db. The user clarified the comment rule in chat: the prohibition targets only NON-FUNCTIONAL inline comments, and chose to keep the constraint-stating exception. Approved explicitly in conversation (option "Mantener excepción").

## Decision
RULE 1 (refined) — No non-functional inline comments.
Code self-explains; narrative/noise inline comments are forbidden. Functional comments are not commentary: toolchain directives (`# noqa`, `// eslint-disable`, `@ts-ignore`, pragmas, coverage markers) are code — permitted, scoped as narrowly as possible, never to mute a warning that should be fixed. The one permitted informative comment states a constraint the code cannot express (non-obvious invariant, external/legal requirement, deliberate trade-off, security warning) — one line, the why. Still forbidden: commented-out code and inline TODO/FIXME (tickets). Docstrings on public APIs stay (interface documentation); design rationale lives in ADRs.

RULE 2 — Test-first for complex logic: unchanged from the superseded decision.

## Rationale
"Comment-free" read as absolute could be taken to ban tool directives, which would break linting/typing workflows. Naming the ban "non-functional inline comments" keeps the intent (no narrative noise) without collateral damage.

## Consequences
developer practices + checklist, and the software-architect ADR-discipline/checklist wording updated to the refined name; qa and security items unaffected.
