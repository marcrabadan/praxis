Three Codex review findings on PR #34, all valid and fixed:

1. (P2) Optional deploy could not skip without a target. The entry gate required `deploy-target-configured`, which (per the workflow schema where gates are pre-start conditions) prevented the no-target case from entering the step to record the skip. Fix: deploy now enters on `release-approved` alone (mirroring the optional `experience` step, which enters on `approved-spec`); the target check is evaluated inside the step — no target -> records "skipped (no deploy target configured)" and advances. gateCriteria for `deploy-target-configured` reworded as an in-step skip-vs-apply decision. Docs updated (artifact-model.md, feature-development.md).

2. (P1) Schema let a production target use promotion:auto (or omit it), contradicting the doctrine that production requires a manual human promotion decision. Fix: added an allOf if/then to each deploy target — environment:"production" => promotion required and const "manual". Verified with jsonschema: prod+auto and prod+missing rejected; prod+manual and staging+auto accepted.

3. (P2) The `deploy` step was unreachable from the canonical /new-feature orchestration (the command listed phases only through release). Fix: added `... -> release -> deploy` to the command's phase chain and `reports/release/deploy-report.md` to its artifact tree, noting it is optional/config-gated. Generated integration prompts regenerated.

Validation: jsonschema conditional tests PASS; make validate-harness OK; make validate-all OK; catalog-check + integrations-check up to date. Relates to accepted decision 20260617-181841-b957.
