Resolves the three open questions from decision 20260617-181841-b957:

1. MCP pins — YES. deploy.md now lists each recommended server with a canonical source + distribution + version floor, and mandates pinning by immutable digest (deploy.mcpServers[*].digest), not a floating tag. Servers: HashiCorp Terraform MCP (official), Kubernetes MCP (containers/kubernetes-mcp-server, community), AWS MCP (awslabs/mcp, official), Azure MCP (Azure/azure-mcp, official), GCP (no single official infra server — prefer the google provider via the Terraform MCP, or a vetted community server). Versions are a floor, confirmed against upstream before wiring/bumping. Config example carries version + digest fields.

2. deploy-report template — YES. Added projects/_template/specs/_template/reports/release/deploy-report.md (status/target/plan/promotion/health/rollback/traceability, DEP-<NNN> id), linked from deploy.md §5.

3. GitOps (Argo/Flux) — NO, not a first-class backend. Recorded as an explicit out-of-scope note in deploy.md §4: valid pull-based alternative, wired by the team in their own repo; this doctrine stays push-based (terraform/kubectl apply via MCP). The deploy report shape is unchanged either way.
